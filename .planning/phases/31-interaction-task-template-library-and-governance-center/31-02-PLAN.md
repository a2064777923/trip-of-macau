<objective>
Add a unified admin governance facade that aggregates experience templates/steps, story overrides, indoor rule references, and reward rule references into one filtered usage/conflict view without replacing existing indoor or reward governance centers.
</objective>

<threat_model>
## T31-02 - Governance Query Safety

Threat: Governance endpoints become unbounded cross-domain scans and degrade the admin backend.

Mitigation:
- All item endpoints must be paginated.
- Filters must be applied before detail enrichment where possible.
- Detail endpoint fetches one deterministic item key.

Threat: Governance output overclaims runtime conflict precision.

Mitigation:
- Findings use deterministic authoring semantics only.
- Each finding includes `findingType`, `severity`, owner refs, and a human-readable explanation.
</threat_model>

<task id="31-02-03" type="backend">
<title>Implement deterministic conflict detection</title>
<read_first>
- packages/admin/aoxiaoyou-admin-backend/src/main/java/com/aoxiaoyou/admin/service/impl/AdminExperienceGovernanceServiceImpl.java
- packages/admin/aoxiaoyou-admin-backend/src/main/java/com/aoxiaoyou/admin/service/impl/AdminExperienceOrchestrationServiceImpl.java
- packages/admin/aoxiaoyou-admin-backend/src/main/java/com/aoxiaoyou/admin/dto/response/AdminExperienceResponse.java
</read_first>
<action>
Add conflict detection to governance service.

Implement these `findingType` values:

- `overlapping_fullscreen_effect`
  - triggered when two or more published items for the same `ownerType + ownerId/ownerCode` have `effectFamily = fullscreen_media` and the same `triggerType`
- `duplicate_reward_grant`
  - triggered when two or more published items in the same owner scope reference the same reward id or reward key in `rewardRuleIdsJson` or reward rule bindings
- `required_step_disabled`
  - triggered when a published override has `overrideMode = disable`, has `targetStepCode`, and targets a required inherited step without replacement
- `shared_pickup_policy_mismatch`
  - triggered when the same pickup or collectible key appears in two item payloads with different effect family or reward policy
- `high_risk_template_without_usage_guard`
  - triggered when a template has `riskLevel in (high, critical)` and is `published`, but has empty `schemaJson` or zero usage

Each conflict response must include `severity`, `findingType`, `title`, `description`, `sourceDomain`, `ownerType`, `ownerId`, `flowId`, `stepId`, `templateId`, `rewardRuleId`, and `itemKey`.
</action>

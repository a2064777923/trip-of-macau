<objective>
Extend the admin backend experience template library from basic CRUD into an operator-usable reusable template library with canonical presets, clone/apply support, and usage visibility. This plan deliberately reuses `experience_templates` and `experience_flow_steps.template_id` instead of creating a parallel template model.
</objective>

<threat_model>
## T31-01 - Template Payload Integrity

Threat: An operator or compromised client submits raw or malformed JSON that later breaks runtime flow compilation.

Mitigation:
- Preserve `validateVersionedJson` for `configJson` and `schemaJson`.
- Preset payloads must include `schemaVersion: 1`.
- Clone endpoints must pass through the same `TemplateUpsert` validation path as create/update.

Threat: Usage endpoints reveal deleted or unrelated data.

Mitigation:
- Usage lookup must filter `deleted = 0` for templates, flow steps, and flows.
- Response must include admin-facing metadata only and no secrets.
</threat_model>

- `templateId`
- `templateCode`
- `templateNameZh`
- `usageCount`
- `flowStepRefs`

- `flowId`

- `flowNameZh`
- `flowType`
- `stepId`
- `stepCode`
- `stepNameZh`
- `stepType`
- `triggerType`
- `status`

Implementation rules:

- Return 404-style `BusinessException` when the template does not exist or is deleted.
- Query only active steps with `deleted = 0`.
- Query only active flows with `deleted = 0`.
- Sort usage refs by `flowId`, then `sortOrder`, then `stepId`.
</action>

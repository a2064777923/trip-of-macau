## 1. System Classification

**System Type:** Hybrid

**Description:**
本 phase 的 AI 系統不是單一聊天機器人，而是一個建在現有 Spring Boot 管理後台與公域後端之上的 AI 控制平面與能力網關。它同時服務兩類用戶：一類是後台編輯與運營成員，用於地圖、POI、故事與媒體資源的 AI 創作輔助；另一類是小程序最終用戶，用於首批可交付的行程推薦規劃與旅行問答。好的結果不是「模型會答」，而是「能力、供應商、密鑰、策略、回退、配額、限流、日誌、資源歷史、最終回填」全都能被後台可靠治理，並與現有 MySQL/COS/admin/public 棧對齊。

**Critical Failure Modes:**
1. API Key、Secret、原始提示詞或敏感輸入被回傳到前端、寫入普通日誌，或泄漏到錯誤響應中。
2. 配額、限流、fallback 或 provider routing 失效，導致異常高併發、成本失控或錯誤模型被調用。
3. AI 生成結果未經候選版本與確認流就直接污染正式資源，或回填到錯誤的表單字段。
4. 行程推薦或旅行問答生成了虛構景點資訊、錯誤開放時間、誤導性交通建議或過度自信答案。
5. 生成歷史、採納歷史與審計日志歸屬錯誤，導致普通後台帳號看到了不該看的其他帳號歷史。

## 1b. Domain Context

> Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Travel operations + story-driven cultural content authoring + smart tourism mini-program runtime

**User Population:**  
- 後台內容編輯、地圖運營、活動運營、超級管理員  
- 小程序旅客，用於行程規劃與旅行問答  

**Stakes Level:** Medium

**Output Consequence:**  
若輸出錯誤，後台側會產生錯誤封面、錯誤 POI 資產、錯誤語音或成本暴衝；前台側會給旅客錯誤行程、錯誤景點資訊或風格失真的 NPC 內容，直接影響品牌可信度、運營效率與真實雲端成本。

### What Domain Experts Evaluate Against

| Dimension | Good (expert accepts) | Bad (expert flags) | Stakes | Source |
|-----------|-----------------------|--------------------|--------|--------|
| 旅遊資訊可信度 | 內容只使用已知景點、路線、故事與配置資料；不確定時明確保留 | 自行虛構地點、交通、營業資訊或歷史背景 | 高 | 產品需求 + 運營場景 |
| 行程可執行性 | 兼顧時間、地理動線、偏好、預算與場景切換，輸出有結構 | 只會寫漂亮文案，沒有順序、時間、備選方案 | 高 | 用戶需求 |
| 編輔資產適配度 | 圖片/語音結果符合資源位類型、比例、透明背景/音色需求，可直接進入候選資產流 | 輸出與資源位不匹配，比例錯、風格跑偏、不可回填 | 高 | 後台編輔需求 |
| 多供應商可治理性 | 能切換、回退、限流、審計、查看用量與成本 | 供應商耦合死在代碼裡，後台改不了 | 高 | AI 能力中心要求 |
| 品牌與敘事一致性 | 故事、POI 介紹、NPC 語氣符合澳門文旅敘事與繁中優先風格 | 語氣機械、文化感薄弱、與既有內容風格割裂 | 中 | 內容運營要求 |
| 成本與濫用控制 | 可按能力/用戶群/時段設限，異常高併發可攔截 | 請求暴衝後才被動發現 | 高 | 運營與平台要求 |

### Known Failure Modes in This Domain

- 景點介紹或問答憑空補完未配置的事實資訊。
- 行程規劃只追求「看起來完整」，忽略動線、時間窗、步行負擔與室內/室外切換。
- POI 疊加圖或城市封面生成不符合透明背景、比例、構圖或資源位要求。
- TTS 對澳門地名、葡語或中葡混合詞彙的發音不穩定。
- 供應商 fallback 後輸出風格、JSON 結構或成本等級突然漂移。
- 日誌收集過度，將完整 prompt、原始輸入、用戶識別信息長期保存，產生隱私與密鑰暴露風險。

### Regulatory / Compliance Context

未從當前 phase 需求中識別到醫療、法律、金融等行業專屬強監管約束，但本 phase 必須遵守以下平台級約束：
- 密鑰只能服務端加密保存，前端不可見明文。
- 提示詞、審計日志與樣本集必須做最小化個資保留，不應長期保存不必要的用戶原始輸入。
- 生成圖片、語音等媒體後續如外部發布，需保留來源、版本與採納記錄，為後續審核流程留口。
- 對接供應商時應啟用其內容安全與用量監控能力；高風險輸入/輸出需要可攔截或人工覆核。

### Domain Expert Roles for Evaluation

| Role | Responsibility |
|------|---------------|
| 內容主編 / 文化策展人 | 評估故事性、歷史敘事、品牌語氣、圖片風格是否可接受 |
| 地圖與空間運營 | 評估 POI / 地圖 / 資源位是否匹配、提示詞模板是否可直接編排 |
| 產品 / 運營負責人 | 校驗能力切分、成本控制、限流與 fallback 策略是否可用 |
| 測試 / QA | 檢查配額、審計、歷史版本、權限隔離、錯誤處理與回填行為 |

## 2. Framework Decision

**Selected Framework:** Spring Boot native AI gateway using `RestClient` + typed capability/provider adapters

**Version:** Spring Boot `3.2.4` / Spring Framework `6.1` `RestClient`

**Rationale:**
- 這個 repo 的 admin/public 後端都已固定在 Spring Boot 3.2.4，且現有外部 HTTP 集成已經在 `WechatAuthService` 中使用 `RestClient`。
- Phase 18 的核心不是 agent orchestration，而是多供應商配置、密鑰治理、策略路由、配額限流、生成歷史、COS 候選/正式資源流，以及後續 public 能力可重用的網關層。
- 用現有 Spring Boot 原生能力實作 typed adapter，可保持與既有 Java DTO、`ApiResponse<T>`、MyBatis-Plus、JWT、COS pipeline 一致，避免為了 AI 引入一套新的 Python/TS 中樞。
- DashScope 官方同時支持 Java DashScope SDK，以及用 OpenAI 官方多語言 SDK 調其 OpenAI 相容接口；因此在架構上最穩妥的是保留「內部統一 adapter 層 + 供應商原生/相容接口分流」。

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| Spring AI `1.x` | 官方 Getting Started 顯示當前 Spring AI 支持 Spring Boot `3.4.x` / `3.5.x`；本 repo 固定在 `3.2.4`，Phase 18 不應強行引入整個後端升版風險 |
| LangGraph | 強項是 Python/TS 的 stateful agent workflow，不適合這個以 Java 管理後台與多供應商治理為主的 phase |
| LangChain / LlamaIndex / CrewAI | 與現有 Java/Spring 主棧不匹配；本 phase 不是先做 RAG 或多 agent，而是先做治理型控制平面 |
| Provider SDK only | 直接把業務綁死在單一 SDK 上，後續切換 OpenAI/Anthropic/DeepSeek/OpenRouter/Minimax/火山引擎會非常痛 |

**Vendor Lock-In Accepted:** Partial

### Core Imports

import org.springframework.web.util.UriComponentsBuilder;
import org.springframework.http.client.SimpleClientHttpRequestFactory;

### Entry Point Pattern
```java
public interface AiCapabilityProvider {
    String providerCode();
    AiNormalizedResult invoke(AiInvocationContext context);
}

public record AiInvocationContext(
        String capabilityCode,
        String scenarioCode,
        Map<String, Object> promptSlots,
        ProviderSecretMaterial secrets,
        AiPolicySnapshot policy
) {}

@Service
public class DashScopeChatProvider implements AiCapabilityProvider {

    private final RestClient restClient;
    private final ObjectMapper objectMapper;

    public DashScopeChatProvider(ObjectMapper objectMapper) {
        SimpleClientHttpRequestFactory requestFactory = new SimpleClientHttpRequestFactory();
        requestFactory.setConnectTimeout(30_000);
        requestFactory.setReadTimeout(30_000);
        this.restClient = RestClient.builder().requestFactory(requestFactory).build();
        this.objectMapper = objectMapper;
    }

    @Override
    public String providerCode() {
        return "dashscope-compatible";
    }

    @Override
    public AiNormalizedResult invoke(AiInvocationContext context) {
        Map<String, Object> body = Map.of(
                "model", context.policy().resolvedModel(),
                "messages", List.of(
                        Map.of("role", "system", "content", context.policy().systemPrompt()),
                        Map.of("role", "user", "content", context.policy().renderedUserPrompt(context.promptSlots()))
                ),
                "temperature", context.policy().temperature(),
                "max_tokens", context.policy().maxTokens()
        );

        DashScopeChatResponse response = restClient.post()
                .uri("https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions")
                .header("Authorization", "Bearer " + context.secrets().apiKey())
                .body(body)
                .retrieve()
                .body(DashScopeChatResponse.class);

        return AiNormalizedResult.fromDashScope(response);
    }
}
```

### Key Abstractions
| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| Capability | 一個可治理的 AI 能力，例如 itinerary planning、travel QA、admin image generation | 能力分頁、限額、供應商綁定、日志統計 |
| Provider Profile | 一個供應商配置快照，含 base URL、secret、模型、timeout、重試 | 發起實際 provider 調用時 |
| Policy Snapshot | 某個能力在某場景下的已解析策略，含模型、fallback、溫度、schema | 真正路由請求前 |
| Invocation Log | 一次 AI 調用的審計記錄，含耗時、用量、成本、狀態、歸屬 | 後台概覽、治理、排錯、歷史 |
| Generation Candidate | 尚未採納的候選圖片/語音結果 | 編輯者預覽、裁剪、回退、重生成 |
| Final Asset Binding | 被編輯者確認後的正式資源關聯 | 回填表單、寫入 `content_assets`、對接 COS |

### Common Pitfalls
1. 把 provider ID、model name、base URL 直接寫死在業務 service 裡，導致後台配置只是擺設。
2. 把圖像生成與長時 TTS 當成普通同步 HTTP 調用，長時間阻塞請求線程，不做 job/polling。
3. 把原始 prompt、完整響應、API Key 或用戶識別信息直接寫進普通應用日志。
4. 讓不同供應商的輸出 JSON 直接滲透到 controller response，導致前端被 provider schema 綁架。
5. 只有全局總限額，沒有能力級、用戶群級、時段級和可疑併發級控制。

## 4. Implementation Guidance

**Model Configuration:**
- 文本對話與結構化輸出：
  - 走 DashScope OpenAI 相容接口，便於後續接入其他兼容 provider。
  - 首批默認能力建議：
    - `旅行問答`: `qwen-plus` 起步，低溫度（`0.2 ~ 0.4`）
    - `行程推薦規劃`: `qwen3.5-plus` 或更高推理等級模型，溫度（`0.4 ~ 0.7`）
- 圖片生成：
  - 走 DashScope 原生圖像生成接口，初始使用 `wan2.6-image`
  - 圖片比例、透明背景、用途詞條由資源位模板決定，不由前端裸傳
- 語音合成：
  - 走 DashScope `SpeechSynthesizer`
  - 預覽優先 `cosyvoice-v3.5-flash`，正式生成可允許升級 `cosyvoice-v3.5-plus`
  - `voice`、`format`、`sample_rate`、`rate`、`pitch`、`word_timestamp_enabled` 必須可策略化

**Core Pattern:**
1. 接收能力請求或編輔工作台請求
2. 解析能力域 / 場景碼 / 資源位類型
3. 載入對應 policy 與 provider config
4. 解密 secret，執行配額、限流、可疑高併發判斷
5. 按能力選擇 provider adapter
6. 將 provider 響應標準化為內部 `AiNormalizedResult`
7. 寫入審計日志、候選版本、成本與延時統計
8. 若為媒體生成，先保存候選資源，再由編輯者確認後進入正式資源流

**Tool Use:**
- 本 phase 不以外部 tool-calling 為主，不引入複雜 agent tool 鏈。
- 旅行問答與行程規劃第一版只允許使用內部受控資料：
  - 後台已發布的城市 / 地圖 / POI / 故事內容摘要
  - 系統設定中的提示詞模板與場景配置
- 禁止直接由模型自由抓網頁資料作最終回答，以避免不可控的時效與可信度問題。

**State Management:**
- MySQL：供應商配置、能力配置、策略、fallback 鏈、配額規則、生成任務、候選歷史、採納歷史、審計日志
- COS：圖片/語音候選結果與正式資產
- Redis（優先在 public path 使用）：熱路徑配額計數、短期防刷、可疑高併發檢測
- 前端本地狀態：只保留工作台草稿，不保留明文 secret

**Context Window Strategy:**
- 旅行問答只注入當前城市 / 地圖 / POI / 故事的精簡摘要，不把整個 CMS 全量灌入 prompt。
- 行程規劃使用「結構化用戶條件 + curated POI/地圖 metadata + 固定輸出 schema」模式。
- AI 創作工作台對圖片/語音只傳入經模板拼裝後的必要字段，不允許無界長 prompt。
- 後台生成圖片時，提示詞由「系統模板 + 表單內容 + 資源位要求 + 編輯者補充」四段組成。

## 4b. AI Systems Best Practices

> Cross-cutting patterns every developer building AI systems needs.

### Structured Outputs with Pydantic

對於行程規劃與旅行問答，先把輸出契約固定，再讓 Java DTO 去對齊它。這裡的 Pydantic 範例用於 eval harness、prompt regression 與 schema contract 設計；實際 runtime 在 Java 端使用等價 DTO / Jackson 驗證。

```python
from typing import List, Optional
from pydantic import BaseModel, Field


class PoiStop(BaseModel):
    poi_code: str = Field(min_length=1)
    title: str = Field(min_length=1)
    start_time: Optional[str] = None
    end_time: Optional[str] = None
    reason: str = Field(min_length=1)


class ItineraryPlan(BaseModel):
    city_code: str
    duration_days: int = Field(ge=1, le=7)
    summary: str
    morning: List[PoiStop] = Field(default_factory=list)
    afternoon: List[PoiStop] = Field(default_factory=list)
    evening: List[PoiStop] = Field(default_factory=list)
    alternatives: List[str] = Field(default_factory=list)
    safety_notes: List[str] = Field(default_factory=list)
```

- Runtime 規則：
  - 模型輸出不符合 schema 時先做一次低成本 retry
  - 第二次仍失敗則切到 fallback policy 或返回結構化錯誤，不把髒數據直接下發
  - 圖片/語音候選結果也要有內部 metadata schema，至少包含 capability、provider、jobId、assetSlot、ownerId、hash

### Async-First Design

- 文本問答與小型結構化規劃可同步調用。
- 圖片生成與較長語音合成必須以 job 模型設計，不長時間佔用 controller thread。
- DashScope 圖像/多媒體場景本身支持同步與異步/流式變體；在本 phase 中：
  - 後台預覽可允許短時同步
  - 正式生成、重試、批量生成、候選資源入庫一律走異步任務
- 常見錯誤：在管理頁點擊一次生成就把瀏覽器請求鎖死 20~60 秒，且沒有 job 狀態、取消、重試與結果回取機制。

### Prompt Engineering Discipline

- system prompt、policy prompt、asset-slot template、editor supplement 必須分層保存，不能全部串成一塊自由文本。
- 後台模板應版本化，支持：
  - 主模板
  - 能力模板
  - 資源位模板
  - 可選 few-shot 示例
- 問答與行程規劃的 prompt 嚴禁直接包含未清洗的長文本或過量表單垃圾字段。
- 模型看不到 secret；模型也不應看到超出當前任務所需的完整內部配置。

### Context Window Management

- 行程規劃：
  - 只傳時間、偏好、預算、城市/地圖範圍、候選 POI 摘要、必要的故事亮點
  - 若 POI 太多，先做 deterministic shortlist，再交給模型排序
- 旅行問答：
  - 只傳與當前問題最相關的 3~10 條受控內容摘要
- 編輔圖片生成：
  - 不傳整個地圖/POI JSON，只傳城市/POI 名稱、介紹摘要、風格模板、比例與輸出要求

### Cost and Latency Budget

- `旅行問答`
  - 目標：`p95 <= 4s`
  - 單次輸出 token 與成本必須落在 policy 可控範圍
- `行程推薦規劃`
  - 目標：`p95 <= 8s`
  - 若輸出需更高質量，優先升級模型，不允許無界重試
- `圖片生成`
  - 目標：後台拿到候選 job id 的交互延時 `< 2s`
  - 真正生成可異步完成
- `語音合成`
  - 短文本預覽 `p95 <= 5s`
  - 正式長文本合成走 job
- 緩存策略：
  - 相同 prompt hash + provider/model + asset slot 的候選生成可去重
  - 旅行問答與行程規劃對相同輸入可做短期結果緩存，但不得跨用戶泄露個性化內容

## 5. Evaluation Strategy

### Dimensions

| Dimension | Rubric (Pass/Fail or 1-5) | Measurement Approach | Priority |
|-----------|---------------------------|----------------------|----------|
| Secret masking | Pass if no API key/secret/plain secret preview/raw prompt leak appears in any admin/public response or standard logs | Code | Critical |
| Provider routing correctness | Pass if request always resolves the expected capability policy, provider, model, and fallback chain | Code | Critical |
| Quota / throttling correctness | Pass if per-capability / per-group / suspicious-concurrency limits fire deterministically | Code | Critical |
| Audit ownership isolation | Pass if admin sees all history but normal operators only see self-owned generation history | Code | Critical |
| Structured output validity | Pass if itinerary / QA outputs conform to schema or fail safely | Code | High |
| Travel answer groundedness | 1-5 based on whether answer stays within configured facts and admits uncertainty when needed | LLM Judge + Human | High |
| Itinerary executability | 1-5 based on realistic ordering, time blocks, alternatives, and relevance to user constraints | Human + LLM Judge | High |
| Asset-slot compliance | Pass if generated image/audio candidate matches requested slot type, ratio, transparency/format constraints | Code + Human | High |
| Prompt template usefulness | 1-5 based on whether prefilled prompt is editable, meaningful, and usually gets near-usable results | Human | Medium |
| Cost / latency adherence | Pass if capability-level SLO and budget thresholds remain within configured band | Code | High |

## 6. Guardrails

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| Secret redaction | 任一管理/公共響應含敏感字段 | Block |
| Policy allowlist | 能力未配置 provider 或 model 不在允許範圍 | Block |
| Quota exceeded | 超出能力/群組/時段配額 | Block |
| Suspicious concurrency | 同一能力/群組短時間高爆發 | Throttle |
| Schema validation failure | 文本結果不符合輸出契約 | Retry then Escalate |
| Provider safety failure | 供應商返回內容安全攔截或高風險錯誤 | Flag |
| Candidate-only asset | 結果尚未被編輯者確認採納 | Flag |

### Offline (Flywheel)

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|----------------------|
| Travel QA groundedness | 每日抽樣成功回答 + 全部 fallback 回答 | 調整 prompt / model / retrieval slice |

## 7. Production Monitoring

**Tracing Tool:** Arize Phoenix as eval/tracing default, with MySQL audit logs as canonical production record

**Key Metrics to Track:**
- provider failure rate
- fallback activation rate
- p95 latency by capability and provider
- cost / token / image / audio usage by capability
- candidate-to-final adoption ratio

**Alert Thresholds:**
- 任一能力 15 分鐘內 provider failure rate `> 5%`
- 任一能力 15 分鐘內 rate-limit / suspicious-concurrency block 次數突然超過基線 `3x`
- 任一能力日成本超過近 7 日均值 `130%`
- 任一 secret masking violation 或 history ownership leak：立即 Sev1
- 旅行問答 / 行程規劃的 schema validation failure rate `> 2%`

**Smart Sampling Strategy:**
- 100% 收集失敗、fallback、被限流、被安全攔截、成本異常高的請求
- 100% 收集正式採納的圖片/語音生成
- 20% 抽樣成功的 itinerary / QA 響應
- 5% 隨機抽樣普通成功調用，用於長期漂移監控

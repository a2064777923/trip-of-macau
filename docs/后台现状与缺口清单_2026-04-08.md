## 二、现状结论摘要

当前后台已不再是空白原型，已经具备：

- 管理员登录
- 仪表盘
- 用户管理
- POI 管理
- 故事线管理（含章节基础 CRUD）
- 运营管理（活动列表）
- 系统管理（奖励 / 配置 / 审计 / 地图资源聚合）
- 测试控制台（模拟定位 / 等级 / 印章 / 重置）

但整体仍属于 **单城市 + 单层内容模型 + 单页聚合式后台**，距离“可支撑正式产品功能的管理底座”仍有明显缺口。

## 三、已验证问题

### 3.1 SPA 路由直连问题仍然存在

已验证：

- 直接访问 `https://macau-trip-2gn2zm5jefa4a987-1301163924.tcloudbaseapp.com/admin/storylines`
- 返回 `404 Not Found`
- 错误为 `Code: NoSuchKey`
- Key 为 `admin%2Fstorylines`

结论：

- 当前静态托管没有处理 SPA history 路由刷新/直连
- 前端当前使用 `BrowserRouter basename="/admin"`
- 若继续使用静态托管目录部署，需要改成 **HashRouter** 或补静态托管重写策略 / 自定义域名反代方案

### 3.2 地图瓦片接口线上仍报错

已验证：

- 调用 `GET /api/admin/v1/system/map-tiles?pageNum=1&pageSize=4`
- 返回：

```json
{
  "code": 5000,
  "message": "服务器开小差了，请稍后再试",
  "data": null
}
```

结论：

- `map-tiles` 链路在线上仍未修复
- 当前代码虽然有 `AdminSystemManagementController#pageMapTiles`、`AdminSystemManagementServiceImpl#pageMapTiles`
- 但运行态仍存在数据库结构、映射、数据异常或运行时兼容问题，需要专项排查

### 3.3 线上访问存在 CloudBase 测试域名风险提示页

已验证：

- 访问 `/admin/login` 时，先出现 CloudBase 测试域名“页面访问提示”中间页

结论：

- 当前后台运行在 CloudBase 默认测试域名
- 不适合作为正式管理后台入口
- 后续需要补充正式域名、别名或受控访问入口说明

## 四、文档层缺口

### 4.1 文档版本与实现不一致

现状：

- `aoxiaoyou-admin-technical-design-v1.0.md` 仍描述为多模块分层后端与传统 ECS/RDS 部署模型
- `后台管理系统技术方案_V1.0.md` 标注“已实现”，但内容仍停留在旧信息架构和旧接口路径
- `aoxiaoyou-admin-api-interface-v1.0.md` 仍使用大量 `/api/v1/*` 路径
- `澳小遊后台管理系统_API设计_V1.1.md` 与当前实际实现更接近，但仍停留在单城市内容系统
- `swagger-template-aoxiaoyou-admin-v1.yaml` 仍是 MVP 范围模板，并未覆盖多城市 / 室内 / AI / 收集系统 / RBAC 完整设计

结论：

- 当前文档体系存在 **多版本并行、接口前缀不统一、部署方式不统一、模块边界不统一** 的问题
- 必须重建一套统一的“正式底座文档体系”并明确以哪套为准

### 4.2 数据库设计文档严重不足

现状：

- `数据库设计.md` 和 `cloudbase_mysql_init.sql` 仅覆盖用户、POI、故事线、奖励、简单 RBAC 与测试控制台
- 不支持多城市、室内地图、AI 导航、收集系统、徽章规则、发布流程、版本回滚

结论：

- 数据库文档必须从“业务表清单”升级为“正式领域模型 + 迁移方案 + 索引策略 + 审计策略”文档

---

## 五、前端信息架构缺口

### 5.1 菜单结构过于扁平

当前一级菜单仅有：

- 仪表盘
- 用户管理
- POI 管理
- 故事线管理
- 运营管理
- 系统管理
- 测试控制台

问题：

- 无法承载多城市 / 空间 / 室内 / AI / 收集系统等复杂业务域
- 同一域内多个子能力被硬塞进单页面，如系统管理混放奖励、配置、审计、地图资源
- 不利于权限分配、菜单权限、数据范围权限与模块化扩展

### 5.2 页面仍以聚合页和表单页为主

现状：

- `SystemManagement` 是多个表格聚合页
- `StorylineManagement` 以单列表 + Drawer + 基础章节表为主
- `POIManagement` 仍是单层 POI 表单
- `OperationsManagement` 仅列表

结论：

- 后台需要重组为带子页的分组式 IA：
  - 地图与空间管理
  - 故事与内容管理
  - 收集与激励管理
  - 用户与进度管理
  - 测试与运营管理
  - 系统与权限管理

### 5.3 缺少画布与图谱型页面

当前没有正式支持：

- 地图画布预览与 POI 图层控制
- 室内楼层切换
- 地图校准页
- 章节分支编排页
- 锚点绑定图谱页
- 权限矩阵页
- AI 配置与测试调试面板

---

## 六、后端领域模型缺口

### 6.1 当前后端仍以 MVP 实体为主

已存在的核心实体大致包括：

- `User`
- `Poi`
- `StoryLine`
- `StoryChapter`
- `Reward`
- `MapTileConfig`
- `SysConfig`
- `SysOperationLog`
- `TestAccount`

问题：

- 不存在正式的 `City` / `MapTileVersion` / `CalibrationPoint` / `Building` / `IndoorFloor` / `IndoorNode` / `IndoorEdge` / `AI Provider` / `Collectible` / `Badge` / `Role-Permission-DataScope` 等领域模型
- 多城市、室内、AI、收集、徽章、推广活动都没有正式建模

### 6.2 缺少正式 RBAC 模型实现

现状：

- 数据库脚本中存在 `sys_admin / sys_role / sys_permission / sys_admin_role / sys_role_permission`
- 但当前前后端实际能力更多停留在“登录 + 角色字段回传”层面
- 后台尚未形成：
  - 管理员列表页
  - 角色配置页
  - 权限点管理页
  - 数据范围页
  - 登录日志页
  - 菜单/按钮/API 权限联动

### 6.3 缺少发布流程与版本管理模型

当前未形成正式模型：

- 草稿 / 待审核 / 已发布 / 已归档 / 已回滚
- 内容版本快照
- 地图版本回滚
- 资源引用关系
- 发布记录与审批记录

---

## 七、API 缺口

### 7.1 当前接口仍偏 CRUD + 列表型

虽已有一定接口基础，但缺少正式产品底座所需接口：

- 城市管理接口
- 城市解锁规则接口
- 地图版本管理接口
- 控制点管理 / 校准接口
- 经纬度转地图坐标接口
- 室内建筑 / 楼层 / 节点 / 路径接口
- AI 供应商配置与测试接口
- 故事分支 / 任务 / 活动章節接口
- 收集物 / 系列 / 徽章 / 规则引擎接口
- 用户城市进度 / 探索进度 / 收集进度 / AI 请求日志接口
- 回滚接口 / 导入导出接口 / 批量处理接口

### 7.2 接口命名与文档不统一

现状：

- 实际实现主要走 `/api/admin/v1/*`
- 部分旧文档仍写 `/api/v1/*`
- OpenAPI 文档使用的 server/base path 与线上部署不一致

结论：

- 需要统一 API 命名规范、错误码体系、分页结构、审计规则、权限点命名

## 九、部署与运行缺口

### 9.1 前端部署方式与路由模式不匹配

现状：

- 前端使用 `BrowserRouter basename="/admin"`
- 静态托管按目录对象存储分发
- 未做 SPA 重写

结果：

- 任意子路由直连 404

### 9.2 后台入口仍是测试域名

问题：

- 存在 CloudBase 风险中间页
- 不是正式管理入口体验
- 不适合面向运营和管理员日常使用

### 9.3 后端运行时兼容性问题未彻底收口

现状：

- 某些接口在线上仍返回 `5000`
- 本地 JDK / 构建环境与 Cloud Run 之间存在差异史
- 需要建立部署前的接口巡检与冒烟测试

## 十一、当前设计假设

为推进后续重构，当前先明确以下假设：

1. 后台管理系统将以 **CloudBase MySQL + Cloud Run + Static Hosting** 为正式基础设施
2. 前端优先采用 **可稳定部署的路由策略**，避免静态托管下继续出现刷新 404
3. 多城市、室内、AI、收集系统不再使用临时字段扩展，而是采用正式领域实体建模
4. 文档将以 `docs/aoxiaoyou-admin-*` 这组英文命名文件为主版本，中文旧版作为对照并逐步统一
